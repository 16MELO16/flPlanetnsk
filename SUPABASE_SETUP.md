# Настройка Supabase для «Цветочной планеты»

Этот документ описывает шаги, которые нужно сделать **один раз** перед тем, как заработает реальная админ-авторизация на сайте.

## 1. Создать проект в Supabase

1. Зайдите на [supabase.com](https://supabase.com) и зарегистрируйтесь (можно через GitHub).
2. Нажмите **New project**.
3. Заполните:
   - **Name**: `flower-planet` (или любое)
   - **Database password**: придумайте надёжный пароль и **сохраните его** (понадобится позже для прямого доступа к БД, но не для сайта)
   - **Region**: ближайший к Новосибирску — `Frankfurt` или `London` (в Supabase нет российских регионов; разница по скорости не критична)
   - **Pricing plan**: Free
4. Нажмите **Create new project** и подождите ~2 минуты, пока проект развернётся.

## 2. Получить URL и публичный ключ

После создания откройте проект и:

1. Слева в меню → **Project Settings** (значок шестерёнки в самом низу).
2. Раздел **API**.
3. Скопируйте две вещи:
   - **Project URL** — выглядит как `https://abcdefghijklmnop.supabase.co`
   - **Project API keys → anon / public** — длинная строка, начинается с `eyJ...`

⚠️ **`anon` ключ безопасно публиковать** в HTML — это его прямое назначение. А вот ключ `service_role` (он там же на странице) **НИКОГДА не вставляйте в сайт** — он даёт полный доступ к БД.

Эти два значения нужно вставить в `index.html` в блок `SUPABASE_CONFIG` (вверху `<script>`).

## 3. Создать админ-пользователя (вас)

1. В Supabase слева → **Authentication** → **Users**.
2. Нажмите **Add user** → **Create new user**.
3. Заполните:
   - **Email**: ваш email (например, `wevvd07@gmail.com`)
   - **Password**: придумайте надёжный пароль (это будет ваш пароль в админку сайта)
   - Поставьте галочку **Auto Confirm User** — иначе придётся подтверждать письмо.
4. **Create user**.

Теперь именно этот email/пароль вы будете вводить в форму входа на сайте.

Если в будущем нужно добавить второго админа (например, флориста) — повторите этот шаг.

## 4. Отключить регистрацию посторонних

По умолчанию Supabase позволяет любому зарегистрироваться и стать пользователем. Для админки этого не нужно — пускаем только тех, кого создали вручную.

1. **Authentication** → **Providers** → **Email**.
2. Найдите **Allow new users to sign up** и **выключите** (toggle off).
3. Сохраните.

После этого `supabase.auth.signUp()` со стороны сайта работать не будет — только вход существующих пользователей.

## 5. Разрешить домен сайта

Чтобы Supabase принимал запросы только с вашего сайта (а не от кого попало):

1. **Authentication** → **URL Configuration**.
2. В поле **Site URL** укажите ваш домен (когда он привяжется к GitHub Pages), например `https://цветочнаяпланета.рф` или `https://flower-planet.ru`.
3. В **Redirect URLs** добавьте тот же URL и для разработки `http://localhost:8080`.
4. Сохраните.

## 6. Что делать дальше

После выполнения шагов 1–5:

1. Откройте `index.html`.
2. Найдите блок `SUPABASE_CONFIG` в начале `<script>` (я его добавлю в коде).
3. Замените `YOUR_SUPABASE_URL` и `YOUR_SUPABASE_ANON_KEY` на значения из шага 2.
4. Сохраните и закоммитьте в GitHub. На GitHub Pages сайт обновится автоматически за 1–2 минуты.

После этого:
- Кнопка **«Войти как админ»** в шапке открывает форму логина.
- После успешного входа появляется кнопка **«Редактировать»** и можно работать с каталогом как раньше.
- При выходе или закрытии браузера админ-режим автоматически снимается.

## 7. Миграция каталога в БД

После того как авторизация заработала, нужно создать таблицу `products` и залить туда начальный каталог. Это превращает сайт из «локального» (каждый видит свою копию) в настоящий магазин с общим каталогом.

**Что нужно сделать:**

1. Откройте Supabase → ваш проект → слева в меню значок **SQL Editor** (выглядит как `</>` или «database»).
2. Нажмите **New query** (или **+** в верхнем углу).
3. Откройте файл `supabase/migrations/001_create_products.sql` в проекте на компьютере, скопируйте **всё его содержимое** целиком.
4. Вставьте в окно SQL Editor.
5. Нажмите кнопку **Run** в правом нижнем углу (или сочетание **Cmd+Enter**).

**Что должно произойти:**

В нижней части окна появится результат — последний запрос в миграции выводит таблицу:

| section   | count |
| --------- | ----- |
| bouquets  | 6     |
| cuttings  | 8     |
| cut       | 8     |
| pot       | 8     |

Итого **30 строк** — все товары из исходного каталога.

**Дополнительно проверить:** слева в меню → **Table Editor** → должна появиться таблица `products`. Откройте её — увидите все 30 товаров с правильными полями.

**Если что-то пошло не так:**

- Ошибка `relation "products" already exists` — таблица уже создана. Это не страшно, миграция написана с `if not exists` и `on conflict do nothing`, можно перезапускать сколько угодно раз.
- Ошибка про права (`permission denied`) — убедитесь, что вы работаете в SQL Editor от имени владельца проекта.
- Любая другая ошибка — пришлите мне точный текст, разберёмся.

После успешной миграции просто перезагрузите сайт (`Cmd+Shift+R`). Каталог теперь читается из БД. Чтобы убедиться — добавьте новый товар через админку, потом откройте сайт в режиме инкогнито (без авторизации): новый товар должен быть виден и там.

## 8. Миграция заказов (мини-CRM)

Делает заказы из корзины настоящими записями в БД с историей и статусами. После миграции в шапке у админа появится 6-я вкладка **«📋 Заказы»** с панелью управления.

**Что нужно сделать:**

1. Supabase → **SQL Editor** → **New query**.
2. Открыть файл `supabase/migrations/002_create_orders.sql`, скопировать всё содержимое (или в терминале: `cat supabase/migrations/002_create_orders.sql | pbcopy`).
3. Вставить в SQL Editor → **Run**.

Внизу появится таблица со списком колонок (id, created_at, customer_name, customer_phone…) — это значит таблица создана. Слева в **Table Editor** должна появиться таблица `orders` (пустая).

**Что меняется на сайте:**

- Когда клиент оформляет заказ через корзину — заказ записывается в `orders` (видно админу) и параллельно отправляется уведомление по email через web3forms (на случай если БД недоступна).
- Админ видит в шапке вкладку **📋 Заказы** с красным бейджем количества новых.
- В панели заказов — фильтры по статусу, селект для смены статуса (новый → в работе → готов → доставлен), внутренние заметки админа, удаление.
- Клиенты вкладку «Заказы» не видят — она появляется только после авторизации.

## 9. Заявки на доступ через Telegram

Эта функция позволяет другому человеку (например, флористу) запросить доступ в админку прямо с сайта. Вы получаете уведомление в Telegram и затем создаёте ему пользователя в Supabase.

Состоит из четырёх шагов: создать бота, узнать свой chat_id, прописать секреты в Supabase, запустить миграцию.

### 9.1. Создать Telegram-бота

1. В Telegram найдите бота **@BotFather** → **Start**.
2. Отправьте команду **/newbot**.
3. BotFather спросит имя бота — например, `Цветочная Планета Уведомления` (это видимое имя).
4. Затем спросит username — например, `flower_planet_notify_bot` (должно заканчиваться на `bot`).
5. BotFather пришлёт сообщение с **токеном**. Выглядит так:
   ```
   8123456789:AAFkdjsufbasdfklhqweuihasdfjlk_xyz123
   ```
   **Скопируйте и сохраните этот токен** — понадобится в шаге 9.3.

### 9.2. Узнать свой chat_id

Бот должен знать, **кому** отправлять уведомления. Это ваш личный chat_id.

1. Откройте созданного бота (в BotFather была ссылка `t.me/...`) → нажмите **Start** или просто отправьте ему сообщение «привет».
2. В браузере откройте ссылку (вставьте свой токен вместо `<TOKEN>`):
   ```
   https://api.telegram.org/bot<TOKEN>/getUpdates
   ```
   Например:
   ```
   https://api.telegram.org/bot8123456789:AAFkdjsufbasdfklhqweuihasdfjlk_xyz123/getUpdates
   ```
3. Откроется JSON. Найдите строку `"chat":{"id":...}`. Например:
   ```json
   "chat":{"id":123456789,"first_name":"Никита",...}
   ```
   Число `123456789` — ваш **chat_id**. Сохраните его.

### 9.3. Прописать секреты в Supabase Vault

Vault — встроенное хранилище секретов в Supabase, чтобы не светить токен в коде.

1. Supabase → **SQL Editor** → **New query**.
2. Вставьте этот SQL, **подставив свой токен и chat_id**:

   ```sql
   -- Удалить старые значения (если уже были)
   delete from vault.secrets where name in ('telegram_bot_token','telegram_admin_chat_id');

   -- Сохранить новые
   select vault.create_secret(
     'ВАШ_ТОКЕН_ОТ_BOTFATHER',
     'telegram_bot_token',
     'Token of @flower_planet_notify_bot'
   );
   select vault.create_secret(
     'ВАШ_CHAT_ID',
     'telegram_admin_chat_id',
     'Chat ID of the owner who receives admin requests'
   );

   -- Проверка: должно показать 2 строки
   select name, description from vault.decrypted_secrets
   where name in ('telegram_bot_token','telegram_admin_chat_id');
   ```

3. Замените `ВАШ_ТОКЕН_ОТ_BOTFATHER` (внутри одинарных кавычек!) на токен из шага 9.1, и `ВАШ_CHAT_ID` на число из шага 9.2.
4. **Run.** Внизу должны появиться 2 строки.

### 9.4. Запустить миграцию

```bash
cd "/Users/nikitamel/Desktop/website flower market"
cat supabase/migrations/003_create_admin_requests.sql | pbcopy
```

Supabase → **SQL Editor** → **+ New query** → Cmd+V → **Run**.

Внизу должна появиться таблица со столбцами `id`, `created_at`, `full_name`, `email`, `reason`, `status`…

### 9.5. Проверить

1. Откройте сайт (`localhost:8080` или ваш домен).
2. Жмёте **🔒 Войти** → внизу модалки ссылка **«Запросить доступ»** → откроется форма.
3. Заполните любым тестовым именем и email → **«Отправить заявку»**.
4. Через 1-2 секунды в вашем Telegram должно прилететь сообщение от бота:
   ```
   🌸 Новая заявка на доступ к админке
   👤 Имя: Тест Тестов
   📧 Email: test@example.com
   ...
   ```
5. На сайте появится зелёное «Заявка отправлена».

Заявка лежит в Supabase → Table Editor → **admin_requests**. Чтобы одобрить — заходите в Authentication → Users → **Add user** с тем же email, придумываете пароль, отправляете его человеку. Чтобы отклонить — удаляете строку из admin_requests.

### Что делать если уведомление не пришло

1. Откройте сайт → DevTools → Console — там не должно быть красных ошибок при отправке заявки.
2. Supabase → Table Editor → admin_requests — заявка должна быть там. Если её нет — проблема в RLS (см. ошибку в Console).
3. Supabase → Database → **Logs** → выберите Postgres. Поищите строки `notify_admin_request_telegram` или `Telegram-секреты`. Если видите `Telegram-секреты не найдены в Vault` — значит секреты в шаге 9.3 не сохранились.
4. Запустите бота вручную: отправьте ему ещё раз сообщение, потом снова через `getUpdates` посмотрите, есть ли ваш chat_id в списке. Если бот молчит — токен скопирован неправильно.

## 10. Фото в Supabase Storage

Без этого шага фото товаров живут в `localStorage` каждого браузера — то есть фото, загруженное с компьютера, не видно на телефоне, и наоборот. После миграции 004 фото будут лежать в **Storage** (облачное хранилище Supabase) и автоматически отображаться у всех посетителей на любом устройстве.

### 10.1. Запустить миграцию

```bash
cd "/Users/nikitamel/Desktop/website flower market"
cat supabase/migrations/004_add_product_photos.sql | pbcopy
```

Supabase → **SQL Editor** → **+ New query** → Cmd+V → **Run**.

Внизу появится одна строка с проверкой: `id=product-photos, public=true, file_size_limit=3145728`. Это значит bucket создан.

### 10.2. Проверить, что bucket появился

Supabase → слева значок **Storage** (выглядит как папка/облако). В списке должен быть бакет **product-photos** — публичный (рядом будет иконка 🌐 или метка `Public`).

### 10.3. Проверить на сайте

1. Откройте сайт → войдите как админ → **✏️ Редактировать**.
2. У любой карточки нажмите **📷** (кнопка в правом верхнем углу) → выберите фото с диска.
3. После «✅ Фото обновлено» откройте этот же сайт **в режиме инкогнито** или **с другого устройства** (например, с телефона).
4. Фото должно отображаться — это значит, оно лежит в Storage и видно всем.

### 10.4. Что со старыми фото

Фото, которые вы уже загрузили **до миграции**, остались в `localStorage` старого браузера и не перенесутся автоматически. Чтобы они стали общими — войдите как админ, нажмите 📷 на каждой такой карточке и **повторно** выберите тот же файл. Сайт сам подхватит, что фото надо положить в Storage.

Альтернатива: можно очистить старые base64 в localStorage командой в DevTools Console:
```js
localStorage.removeItem('fp_photos')
```
после этого все «локальные» фото исчезнут, и вы увидите, у каких карточек ещё нет фото в БД.
